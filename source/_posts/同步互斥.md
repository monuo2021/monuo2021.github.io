在操作系统中，不同核心上的多个进程(线程)可能会协作完成同一个任务，导致对共享数据的竞争。为了辅助多个进程(线程)之间进行同步，保证程序的正确性，操作系统提供了一系列同步原语供应用使用。

## 一、并发术语

### 1.1 临界区

临界区(critical section)是访问共享资源的一段代码，资源通常是一个变量或数据结构。

### 1.2 竞态条件

竞态条件(race condition)出现在多个执行线程大致同时进入临界区时，它们都试图更新共享的数据结构，导致了我们不希望的结果，比如数据覆盖。

### 1.3 不确定性

不确定性(indeterminate)程序由一个或多个竞态条件组成，程序的输出因运行而异，具体取决于哪些线程在何时运行。这导致结果不是确定的，而我们通常期望计算机系统给出确定的结果。

### 1.4 同步原语

为了避免这些问题，线程应该使用某种互斥（同步）原语，这样做可以保证只有一个线程进入临界区，从而避免出现竞态，并产生确定的程序输出。下图是对同步原语的总结，从这张图中可以看到同步方法的整体框架。

## 二、同步机制遵循的原则

无论是硬件方法实现同步互斥，还是软件方法，亦或者是高级抽象的方法，都必须遵循以下原则：

- **空闲让进**

  当没有进程在临界区时，任何进程可进入以有效利用资源。

- **忙则等待**

  当有进程在临界区，其他进程不能进入。

- **有限等待**

  等待进入临界区的进程，不能无限期的等待，以免陷入“永远等待”。

- **让权等待（可选）**

  进程不能进入临界区时，立即释放 CPU，以免 CPU 陷入“忙等”。

## 三、互斥锁

### 3.1 软件实现：皮特森算法

![image-20211113141013741](https://gitee.com/tanliang-TL/blogpic/raw/master/img/image-20211113141013741.png)

在这个算法中有两个重要的变量：

- **全局**数组 flag

  有两个布尔成员 flag[0] 和 flag[1]，分别代表线程 0 与线程 1 是否尝试进入临界区。

- **全局**变量 turn

  如果两个线程都申请进入临界区，turn 将决定最终进入临界区的线程。如上图，线程 0 和线程 2 都申请进入临界区，即 flag[0] 和 flag[1] 都为 true； 如果线程 0 先执行 turn = 1，线程 1 后执行 turn = 0，则 turn 最终等于 0，此时线程 0 进入临界区。

### 3.2 原子操作

原子操作是指**不可被打断**的一个或一系列的操作。也就是说，这一系列指令相当于一条指令，**要么都执行完，要么就一条也不执行**。

那么如何实现原子操作？

首先我们要知道，靠编译器自动生成原子操作不太可能。

1. 编译器并不能检测出哪个变量需要原子操作；
2. 编译器必须要考虑代码的移植性。有些硬件平台支持原子操作的机器指令，有的硬件平台不支持。

因此，我们需要其他方法实现原子操作。在 x86 平台有很多原子指令，比如原子加、原子减，原子读写等。我们可以使用这些指令，用在 C 函数中嵌入汇编代码就写出需要的原子操作函数。

例如**比较与置换（CAS）**原子操作：

```c
int atomic_CAS(int *addr, int expected, int new_value)
{
  asm volatile("lock cmpxchg %[new], %[ptr]"
		:"+a"(expected), [ptr] "+m"(*addr)
    :[new] "r"(new_value)
    :"memory");
  return expected;
}
```

值得注意的是，原子操作只有**单个变量全局数据**的情况下，这种变量非常实用，如全局计数器、状态标志变量等。当要**操作的数据很多**的情况下，用原子操作就不适合。

### 3.3 硬件实现：关闭中断

![image-20211113123344198](https://gitee.com/tanliang-TL/blogpic/raw/master/img/image-20211113123344198.png)

那么关闭中断是否满足同步机制的原则呢？关闭中断可以防止执行临界区的线程被抢占，满足**忙则等待**；而**有限等待**依赖于内核中的调度器，如果能在有限时间内调度到该线程，则该线程就可以在有限时间内进入临界区，满足**有限等待**；每个线程离开临界区时都开启了中断，允许调度器调度其他线程执行，从而达到了**空闲让进**的要求。

当然，关闭中断并不能解决所有的临界区问题。在上图可知，关闭中断可以解决单 CPU 核的临界区问题。但如果是**多核处理器**，关闭中断不能阻塞其他进程的执行。

### 3.4 互斥锁

由于关闭中断只能解决单核处理器的临界区问题，而现代处理器基本都是多核处理器，因此我们需要新的机制来解决同步互斥问题。

利用硬件保证的原子操作，我们可以实现互斥锁来解决。顾名思义，互斥锁类似于锁，有 lock 和 unlock 两种操作，并且一把锁在同一时刻只能被一个线程拥有。当线程试图访问临界区时，操作系统会给它加锁（lock），只有拥有锁的线程才能执行临界区的代码。在线程离开临界区时，线程会释放锁（unlock）。

互斥锁种类有很多，下面介绍两种互斥锁：**自旋锁和排号锁**。

#### 3.4.1 自旋锁

下面两图时自旋锁的原理和实现方法，原理较为简单，实现也不是太难，不细讲。要点就是利用原子操作 CAS 判断 lock 是否空闲（是否为 0），空闲则加锁（lock = 1），否则就一直循环。

<img src="https://static001.geekbang.org/resource/image/61/88/619c27c6400344db2310fb82ce8d5788.jpg?wh=2195*2405" alt="img" style="zoom: 20%;" />

```c
void lock_init(int *lock)
{
  *lock = 0;
}

void lock(int *lock)
{
  while(atomic_CAS(lock, 0, 1) != 0)
    ;
}

void unlock(int *lock)
{
  *lock = 0;
}
```

知道自旋锁原理和实现方法后，来看看它是不是满足同步机制应遵循的原则。首先很容易知道，自旋锁满足空闲让进、忙则等待（分析方法如关闭中断）。但自旋锁并不能保证**有限等待**。从实现可以看出，自旋锁获取锁成功与否主要取决于原子指令 CAS，而原子指令的成功与否取决于硬件特性。如果在异构环境，小核由于运行频率低，在与大核竞争时成功完成原子操作的概率远小于大核。因此运行在小核的线程永远都获取不到锁。

#### 3.4.2 排号锁

为了解决自旋锁的不公平性，于是产生了排号锁。排号锁采用了一种更加公平的选择策略：按照竞争者申请锁的顺序传递锁。因此可以认为，排号锁是一个先进先出（FIFO）的等待序列。

```c
struct lock
{
  volatile int owner;
  volatile int next;
}

void lock_init(struct lock *lock)
{
  lock->owner = 0;
  lock->next = 0;
}

void lock(struct lock *lock)
{
  // 拿取自己的序号
  volatile int my_ticket = atomic_FAA(&lock->next, 1); // FAA是原子操作——获取并累加
  while(lock->owner !=  my_ticket)
    ;
}

void unlock(struct lock *lock)
{
  // 传递给下一个竞争者
  lock->owner++;
}
```

## 四、条件变量

对线程同步而言，有两点很重要（从同步机制遵循的原则可以看出）：

- 互斥访问
- 等待

互斥锁用于解决临界区问题，保证互斥访问共享资源。当共享资源被某一线程占用时，其他线程会陷入**循环等待**。无意义的循环等待不仅浪费 CPU 资源，还增加了系统的能耗。我们可以采用一种机制，在没有资源时将线程挂起，在有资源时将其唤醒。而条件变量就是我们需要的这种机制。

#### 4.1 条件变量的实现

```c
struct cond {
  struct thread *wait_list;
}

void cond_wait(struct cond *cond, struct lock *mutex)
{
  list_append(cond->wait_list, thread_self());
  atomic_block_unlock(mutex);	// 原子挂起并放锁
  lock(mutex);	// 重新获得互斥锁
}

void cond_signal(struct cond *cond)
{
  if(!list_empty(cond->wait_list))
    wakeup(list_remove(cond->wait_list));
}

// 唤醒所有等待在条件变量的线程
void cond_broadcast(struct cond *cond)
{
  while(!list_empty(cond->wait_list))
    wakeup(list_remove(cond->wait_list));
}
```

> 互斥锁 VS 条件变量
>
> 互斥锁用于解决临界区问题，保证互斥访问共享资源。
>
> 条件变量通过提供挂起/唤醒机制来避免循环等待，节省 CPU 资源。条件变量需要和互斥锁搭配使用。

## 五、信号量

以我目前的理解，**信号量 = 互斥锁 + 条件变量**。

一个信号量 S 是个整型变量，它除了初始化外只能通过两个标准原子操作：wait () 和 signal() 来访问：

- 操作 wait() 最初称为 P（荷兰语 proberen，测试）；
- 操作 signal() 最初称为 V（荷兰语 verhogen，增加）；

### 5.1 信号量语义

```c
void wait(int *s)
{
  while(*s < 0)
    ;	// 循环忙等
  *s = *s -1;
}

void signal(int *s)
{
  *s = *s + 1;
}
```

wait 用于等待，当**信号量的值（其代表资源数量）**小于或等于 0 时进入循环等待。只有信号量的值大于 0 时，才停止等待并消耗该资源（第五行，减少信号量的值）。

signal 用于通知，它会增加信号量的值供 wait 的线程使用。

> 注意：上面代码只是为了解释信号量的语义。如果想要保证多个线程并发执行的正确，需要添加很多东西。

### 5.2 信号量实现

目前我看到的信号量实现有两种：

- 只使用互斥锁
- 使用了互斥锁和条件变量

#### 5.2.1 基于互斥锁的信号量实现

```c
struct sem {
  int value;
  struct process *list;
}

void wait(struct sem *S)
{
  S->value--;
  if (S->value < 0) {
    add this process to S->list;
    block();
  }
}

void signal(struct sem *S)
{
  S->value++;
  if (S->value <= 0) {
    remove a process P from S->list;
    wakeup(P);
  }
}
```

#### 5.2.2 基于互斥锁和条件变量的信号量实现

```c
struct semaphore {
  int value;	// 无线程等待时，代表剩余的资源数量；有时，其绝对值代表正在等待的线程数量
  int wakeup;	// 有线程等待时可用的资源数量
  struct lock sem_lock;
  struct cond sem_cond;
}

void wait(struct sem *S)
{
  lock(&S->sem_lock);
  S->value--;
  if (S->value < 0) {
    do {
			cond_wait(&S->sem_cond, &S->sem_lock);
    } while(S->wakeup == 0);
    S->wakeup--;
  }
  unlock(&S->sem_lock);
}

void signal(struct sem *S)
{
  lock(&S->sem_lock);
  S->value++;
  if (S->value <= 0) {
    S->wakeup++;
    cond_signal(&S->sem_cond);
  }
  unlock(&S->sem_lock);
}
```

两种实现方式疏导同归，本质上就是一样的。在之后的同步问题——生产者消费者问题那里我们可以看到两者的异同。

## 八、管程

管程是一种用于多线程互斥访问共享资源的程序结构

- 采用面向对象方法,简化了线程间的同步控制
- 任一时刻最多只有一个线程执行管程代码
- 正在管程中的线程可临时放弃管程的互斥访问,
- 等待事件出现时恢复

管程的使用

- 在对象/模块中,收集相关共享数据
- 定义访问共享数据的方法

## 九、同步带来的问题

### 9.1 死锁

当有多个线程为有限资源竞争时，有点线程就会因为再某一时刻没有空闲的资源而陷入等待。**死锁**就是指这一组中的每一个线程都在等待组内其他线程释放资源从而导致的无线等待。

```c
void proc_A(void)
{
  lock(A);
  // t0时刻
  lock(B);
  // 临界区
  unlock(B);
  ublock(A);
}

void proc_B(void)
{
  lock(B);
  // t0时刻
  lock(A);
  // 临界区
  unlock(A);
  ublock(B);
}
```

在 t0 时刻，线程 A 获得了锁 A，并试图获取锁 B；同一时刻，线程 B 获取了锁 B，并试图获取锁 A。由于两线程都持有对方想要的资源，但两者都不能进入临界区继续执行，从而陷入无限等待。

#### 9.1.1 死锁产生的必要条件

- **互斥访问**

- **持有并等待**

  线程持有一些资源，并等待一些资源。

- **资源非抢占**

  一旦一个资源被持有，除非持有者主动放弃，否则其他竞争者都得不到这个资源 。

- **循环等待**

  指存在一系列线程 T0、T1、T2、...... 、Tn，其中 T0 等待 T1，T1 等待 T2，...... ，Tn 等待 T0。因此形成一个循环。由于循环中任一线程都不能等待资源，因而不能释放已占有的资源，导致了循环等待。

#### 9.1.2 死锁解决方法

- **出问题再解决：死锁的检测与恢复**

  当死锁出现时，需要一个第三者来打破僵局，操作系统就扮演这个角色。

  ![image-20211114100017507](https://gitee.com/tanliang-TL/blogpic/raw/master/img/image-20211114100017507.png)

- **设计时避免：死锁预防**

  - 避免互斥访问：通过其他手段(如代理执行)
  - 不允许持有并等待：一次性申请所有资源
  - 资源允许抢占：需要考虑如何恢复
  - 打破循环等待：按照特定顺序获取资源
    - 所有资源进行编号
    - 所有进程递增获取
    - 任意时刻：获取最大资源号的进程可以继续执行,然后释放资源
